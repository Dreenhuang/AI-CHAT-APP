第1章：产品经理的角色定位与核心价值

1.1 产品经理到底是什么？

一句话定义：产品经理是产品的"CEO"，负责产品的全生命周期管理，连接用户、技术和商业。

1.1.1 产品经理的三重身份

身份 核心职责 具体工作
用户代言人 理解用户需求，代表用户利益 用户调研、竞品分析、需求优先级排序
技术翻译官 将业务需求转化为技术可实现的功能 编写PRD、与技术团队沟通、验收测试
商业操盘手 对产品的商业结果负责 数据分析、产品迭代、商业化策略

1.2 产品经理的核心能力模型

1.3 新人产品经理常见的5个误区

❌ 误区1："产品经理就是画原型的"

真相：原型只是表达工具，不是核心价值。真正的价值在于思考过程和决策质量。优秀的产品经理80%时间在思考，20%时间在输出。

❌ 误区2："我要满足所有用户的需求"

真相：试图满足所有人 = 满足不了任何人。必须学会说"不"，并给出合理理由。优先级排序是产品经理最重要的能力之一。

❌ 误区3："技术细节不需要了解"

真相：不需要会写代码，但必须理解技术可行性。不懂技术会被开发"忽悠"，导致需求无法落地。

第3章：需求分析方法论

3.1 需求的四个层次（Kano模型）

层次 特征 处理策略 示例
兴奋型 "Wow！没想到还有这个" 资源允许时做 AI智能推荐、个性化定制
期望型 "这个功能应该要有吧" 尽量做好 页面加载速度、操作流畅度
基本型 "没有就不行了" 必须做 登录功能、数据保存
隐含需求 用户没说但真正需要的 深入挖掘 未表达的痛点

3.2 需求优先级排序方法

MoSCoW法优先级分布 推荐比例

Must have（必须有）: 60% - 本期必须完成

Should have（应该有）: 20% - 重要但可延期

Could have（可以有）: 15% - 锦上添花

Won't have（不会有）: 5% - 明确不做

RICE评分公式：RICE Score = (R × I) / (E × D)

R=影响用户数 I=影响程度 E=工作量 D=置信度

第4章：PRD文档编写完整指南

4.1 什么是PRD？

PRD = Product Requirements Document（产品需求文档）

PRD是产品经理最重要的输出物，它是产品愿景的蓝图，是开发团队的施工图，是验收测试的标准。

4.2 PRD编写的5个黄金法则

1️⃣

一个功能一个章节

每个功能独立成节，便于追踪和管理

2️⃣

用表格替代大段文字

结构化表达，提升可读性和可执行性

3️⃣

必须包含异常流程

不只写Happy Path，要覆盖所有Edge Cases

4️⃣

每个功能都要有验收标准

功能+性能+安全+兼容性四维验收

5️⃣

保持可追溯性

每个操作都建立完整的追溯链路

第6章：前后端协作的黄金法则

6.1 API First 推荐模式 ⭐

核心理念：API即契约（Contract），前后端共同遵守同一份API文档，并行开发，最后快速集成。

✅ API First 的优势
  • 前后端并行开发，缩短整体周期
  • Mock Server模拟真实数据，前端无需等待
  • API文档作为"合同"，减少扯皮
  • 联调变成验证，而非调试

第四部分：管理后台专题篇

第10章：为什么AI写的PRD总是忽略管理后台？

⚠️ 三大根因
  1. AI训练偏差：公开数据主要是用户端产品，管理后台闭源
  2. PM思维盲区：只关注用户端，认为后台"很简单就是CRUD"
  3. 低估复杂度：实际管理后台包含权限系统、审计日志、批量操作等高级功能
✅ 解决方案
  1. 方案一：PRD阶段就纳入管理后台需求Checklist
  2. 方案二：双轨并行开发（用户端+管理后台同时启动）
  3. 方案三：建立管理后台模板库（CRUD/审核流/权限管理等标准模板）

第11章：管理后台设计的完整方法论

11.1 四大设计原则

原则 核心思想 具体实践
效率优先 管理员每天重复操作数百次 减少点击次数、键盘快捷键、批量操作
信息密度适中 太密眼花缭乱，太疏效率低下 每行高度48-56px，列宽比例合理分配
一致性 同样的功能同样的交互 布局、颜色、文案风格统一
反馈及时 每个操作都有反馈 Toast提示、Loading状态、二次确认

第12章：权限系统与角色设计

12.1 RBAC权限模型

RBAC = 基于角色的访问控制

用户 → 角色 → 权限（用户通过拥有角色来获得权限）

12.2 本项目三角色设计

角色 中文名称 适用人群 权限范围
SUPER_ADMIN 超级管理员 产品负责人/技术负责人 所有权限，包括系统配置和用户管理
OPERATOR 运营人员 内容运营/客服 内容管理、数据查看、审核操作
VIEWER 只读查看者 实习生/外部顾问 仅查看数据，不能修改
🛡️ 权限控制三层次
  1. UI层：隐藏无权限按钮（用户体验优化）
  2. API层：拦截无权限请求（真正安全控制）⭐ 核心
  3. 数据层：过滤无权访问数据（数据安全）

重要提醒：前端权限控制只是为了用户体验，真正的安全必须靠后端实现！
