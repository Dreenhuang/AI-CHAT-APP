## 一、项目概述与目标

### 1.1 背景说明

当前PRD辩论系统实现了基础的"提案者→审查者"线性讨论模式，本质上是**PRD文档评审**的流程。但根据实际使用场景，用户需要更多样化的讨论形式来适配不同场景：

- 一对一深度商量
- 多人圆桌研讨
- 正式辩论对抗
- 结构化决策
- 头脑风暴
- 多AI协同

### 1.2 四大核心任务

| 任务编号 | 任务内容 | 优先级 |
|---------|---------|--------|
| 任务一 | 现有PRD评审模式固化 + 新增16种讨论模式 | P0 |
| 任务二 | 设计30+ Soul角色库，支持模式自动匹配 | P0 |
| 任务三 | 用户自定义输出字数（简短/深入/详细） | P1 |
| 任务四 | 输出显示样式优化 | P1 |

### 1.3 核心设计原则

1. **不破坏现有逻辑** - 现有PRD评审模式完整保留
2. **配置驱动** - 新模式通过配置定义，代码逻辑统一
3. **角色池化** - 30+角色作为可复用资源
4. **渐进增强** - 分阶段实施，逐步上线

## 二、PRD评审模式固化

### 2.1 模式定义

**模式名称**：报告PRD评审模式
**模式编号**：`prd-review`
**模式分类**：分工专项研讨式（变体）
**参与角色**：主持人、提案者、审查者（固定3人）

### 2.2 现有逻辑梳理（必须完整保留）

```
阶段 (Phase) × 轮次 (Round) 双重循环：

第一阶段：需求探查 (Probe)
  第1轮：提案者提出方案 → 审查者质疑/审查
  第2轮：提案者回应质疑 → 审查者继续追问
  第3轮：提案者补充方案 → 审查者验证
  第4轮：提案者完善方案 → 审查者确认
  第5轮：进入下一阶段

第二阶段：方案设计 (Design)
  [同上5轮循环]

第三阶段：实现规划 (Implementation)
  [同上5轮循环]

第四阶段：验证确认 (Validation)
  [同上5轮循环]

第五阶段：[可选扩展阶段]
  [同上5轮循环]
```

### 2.3 每轮内部流程（代码级别保留）

```javascript
// 每轮流程 - 必须完整保留
const PRD_REVIEW_ROUND_FLOW = {
  // 1. 提案者发言（完整方案输出）
  step1: {
    actor: 'proposer',
    action: 'proposal',
    label: '提出方案/需求/设计',
    description: '完整呈现核心内容，不追求完美'
  },

  // 2. 审查者质疑（挑漏洞、找问题）
  step2: {
    actor: 'reviewer',
    action: 'question',
    label: '质疑审查',
    description: '客观指出问题、漏洞、风险，不否定整体'
  },

  // 3. 提案者回应（解释/修正）
  step3: {
    actor: 'proposer',
    action: 'respond',
    label: '回应质疑',
    description: '有理由则解释，有漏洞则修正'
  },

  // 4. 审查者再审（补充追问）
  step4: {
    actor: 'reviewer',
    action: 're-review',
    label: '再次审查',
    description: '确认问题是否解决，继续补充'
  },

  // 5. 共同确认（进入下一轮或下一阶段）
  step5: {
    actor: 'all',
    action: 'confirm',
    label: '确认/记录',
    description: '记录承诺(Commitment)，确认本轮完成'
  }
};
```

### 3.1 一对一双向商量类（2方参与）

#### 模式1：双向自由对谈式

**基本信息**：
- 模式编号：`free-dialogue`
- 模式名称：双向自由对谈式
- 参与人数：2人
- 是否需要主持人：否
- 适配场景：两人简单交换想法、初步探讨

**角色配置**：
```javascript
{
  id: 'free-dialogue',
  name: '双向自由对谈式',
  category: '一对一双向商量',
  minRoles: 2,
  maxRoles: 2,
  defaultRoles: [
    { roleType: 'participant-a', name: '甲方', description: '参与方A' },
    { roleType: 'participant-b', name: '乙方', description: '参与方B' },
  ],
  rolesAreEqual: true,  // 双方平等，无主次
}
```

**讨论流程**：
```
顺序1：甲方完整输出
  → 甲方不受干扰地完整表达自身观点、方案、认知
  → 时长/字数：由输出深度设置决定

顺序2：乙方完整输出
  → 乙方完整表达与甲方不同的观点、补充、质疑
  → 时长/字数：由输出深度设置决定

顺序3：双方商讨
  → 针对对方观点进行讨论
  → 可以追问、反驳、补充
  → 循环直到双方达成初步共识

顺序4：共同结论
  → 整合双方观点，形成共同认可的结论
  → 记录讨论结果
```

**流程图示**：
```
[甲方输出] → [乙方输出] → [双方商讨] ↔ [共识]
                                        ↓
                              [输出共同结论]
```

**Prompt模板**：
```javascript
// 甲方输出Prompt
`【甲方角色】
你叫{name}，你的性格是{character}。
请完整输出你对这个议题的看法和方案：
议题：{topic}
要求：完整表达，不被打断

【字数要求】
{lengthRequirement}
`

// 乙方输出Prompt
`【乙方角色】
你叫{name}，你的性格是{character}。
甲方刚才说了：{甲方内容}
请完整输出你的看法和方案，可以补充、质疑、反对甲方观点。

【字数要求】
{lengthRequirement}
`

// 商讨Prompt
`【讨论环节】
甲方观点：{甲方内容}
乙方观点：{乙方内容}
请双方针对对方观点进行商讨。
{questionPrompt}

【字数要求】
{lengthRequirement}
`
```

#### 模式2：一问一答追问式

**基本信息**：
- 模式编号：`qa-chase`
- 模式名称：一问一答追问式
- 参与人数：2人
- 是否需要主持人：否
- 适配场景：精准答疑、缩小信息差

**角色配置**：
```javascript
{
  id: 'qa-chase',
  name: '一问一答追问式',
  category: '一对一双向商量',
  minRoles: 2,
  maxRoles: 2,
  defaultRoles: [
    { roleType: 'asker', name: '提问方', description: '主动提问，寻求解答' },
    { roleType: 'answerer', name: '回答方', description: '回答问题，补充信息' },
  ],
}
```

**讨论流程**：
```
顺序1：提问方提问
  → 提出一个具体问题
  → 问题应该清晰、具体、可回答

顺序2：回答方回答
  → 直接回答问题
  → 提供事实、数据、解释

顺序3：提问方追问
  → 基于回答继续追问
  → 挖掘更深层次的信息

顺序4：回答方再答
  → 回应追问
  → 继续深入

[循环] 继续追问直到：
  → 信息差消除
  → 或达到设定的追问轮次上限

顺序N：共识确认
  → 确认已获得满意答案
  → 记录结论
```

**流程图示**：
```
[提问] → [回答] → [追问] → [再答] → [追问] → [再答]
                                              ↓
                                    [信息差消除/达轮次上限]
                                              ↓
                                       [记录结论]
```

**特殊控制参数**：
```javascript
{
  maxFollowUps: 5,        // 最大追问轮数
  questionClarity: true,  // 要求问题清晰具体
  answerDepth: 'auto',    // 自动深度（根据回答质量调整）
}
```

#### 模式3：互补完善式

**基本信息**：
- 模式编号：`complement`
- 模式名称：互补完善式
- 参与人数：2人
- 是否需要主持人：否
- 适配场景：整合双方优势、补充漏洞

**角色配置**：
```javascript
{
  id: 'complement',
  name: '互补完善式',
  category: '一对一双向商量',
  minRoles: 2,
  maxRoles: 2,
  defaultRoles: [
    { roleType: 'presenter', name: '方案方', description: '提出初始方案' },
    { roleType: 'supplementer', name: '补位方', description: '补充漏洞盲区' },
  ],
  rules: {
    // 补位方不能重复方案方内容
    supplementerMustNotRepeat: true,
    // 补位方不能否定整体
    supplementerMustNotDeny: true,
  }
}
```

**讨论流程**：
```
顺序1：方案方完整输出
  → 完整呈现核心方案/结论
  → 不遗漏主要内容
  → 明确自身优势

顺序2：补位方补充漏洞
  → 只补充方案方未覆盖的：
    - 漏洞
    - 盲区
    - 细节
    - 备用方案
  → 不重复方案方内容
  → 不否定整体框架

顺序3：方案方整合
  → 吸收补位方的补充
  → 优化自身方案
  → 整合双方思路

顺序4：补位方再审
  → 审阅优化后的方案
  → 若有遗漏继续补充
  → 若无遗漏确认通过

顺序5：共同敲定
  → 协商取舍（如有冲突）
  → 优先核对信息真实性
  → 达成一致结论
```

**核心规则强化**：
```javascript
// 补位方必须遵守的规则
const supplementRules = [
  '不重复方案方已说的内容',
  '不否定整体框架和核心结论',
  '只补充：漏洞、盲区、细节、备用方案',
  '补充内容必须有依据',
];
```

### 3.2 多人圆桌合议类（3方及以上）

#### 模式4：圆桌自由研讨式

**基本信息**：
- 模式编号：`roundtable-free`
- 模式名称：圆桌自由研讨式
- 参与人数：3-10人
- 是否需要主持人：需要（协调）

```javascript
    { roleType: 'member', name: '成员1', description: '参与讨论' },
    { roleType: 'member', name: '成员2', description: '参与讨论' },

  ],
  // 可选：增加专家角色
  optionalExpertRoles: [
    { roleType: 'expert-tech', name: '技术专家' },
    { roleType: 'expert-business', name: '业务专家' },
    { roleType: 'expert-user', name: '用户专家' },
  ]
```

**讨论流程**：
```
顺序1：主持人开场
  → 介绍议题背景
  → 说明讨论目标
  → 宣布讨论规则（无批评、自由发言）

顺序2：自由发言（多轮）
  → 所有人自由发言
  → 可以随时补充他人观点
  → 可以提出全新角度
  → 主持人协调发言秩序
  → 直到观点充分发散

顺序3：观点归类
  → 主持人将所有观点归类整理
  → 去除重复
  → 识别核心观点

顺序4：收敛确认
  → 针对归类后的观点进行投票/确认
  → 筛选出最重要观点
  → 形成结论框架

顺序5：结论输出
  → 主持人汇总
  → 全员确认
```

#### 模式5：轮值发言合议式

**基本信息**：
- 模式编号：`rotating-speaker`
- 模式名称：轮值发言合议式
- 参与人数：3-10人
- 是否需要主持人：需要
- 适配场景：确保每个人都被听到

**角色配置**：
```javascript
{
  id: 'rotating-speaker',
  name: '轮值发言合议式',
  category: '多人圆桌合议',
  minRoles: 3,
  maxRoles: 10,
  defaultRoles: [
    { roleType: 'host', name: '主持人', description: '控场轮次' },
    { roleType: 'member', name: '成员', description: '轮值发言' },
  ],
  rotationRules: {
    sequential: true,        // 顺序发言
    eachSpeaksOnce: true,    // 每人一轮只说一次
    continueUntilConsensus: true,  // 循环直到共识
  }
}
```

**讨论流程**：
```
顺序1：主持人说明议题
  → 明确讨论主题
  → 说明轮次规则
  → 宣布第一轮开始

顺序2：第一轮轮值发言
  → 成员1发言（完整表达观点）
  → 成员2发言（完整表达观点）
  → 成员3发言...
  → 所有人轮流发言一次
  → 其他人聆听，不打断

顺序3：第二轮轮值发言
  → 基于第一轮观点进行回应
  → 补充自己观点
  → 继续顺序发言

顺序4：第N轮轮值发言
  → 循环直到：
    - 共识形成
    - 或达到最大轮次

顺序5：主持人汇总
  → 整理各轮发言
  → 形成结论
  → 全员确认
```

**轮次控制**：
```javascript
{
  maxRounds: 5,           // 最大讨论轮数
  allowFreeDiscussion: false,  // 是否允许自由讨论
  speakingTime: 'auto',   // 自动控制发言时长
}
```

**基本信息**：

- 参与人数：4人及以上（1主持人 + N维度方）

**角色配置**：

```javascript
{
  id: 'split-thesis',

    { roleType: 'dimension-2', name: '维度2方', description: '覆盖维度2' },
    { roleType: 'dimension-3', name: '维度3方', description: '覆盖维度3' },
    // 可扩展更多维度
  ],
  dimensions: ['技术可行性', '成本效益', '用户体验', '风险控制'],
```

**讨论流程**：
```
顺序1：主持人分配维度
  → 将核心议题拆分为多个不重复、无遗漏的维度
  → 每个维度分配给一个参与方
  → 明确每个维度的讨论重点

顺序2：各维度方独立输出
  → 各方独立思考
  → 专注自身维度
  → 完整输出该维度的观点
  → 不涉及其他维度

顺序3：互相审阅
  → 各方了解其他维度的观点
  → 检查是否有漏洞
  → 检查是否有逻辑冲突

顺序4：补充修正
  → 各方针对他人的内容
  → 提出补充、修正建议
  → 聚焦自身维度的优化点

顺序5：主持人汇总整合
  → 整合所有维度的内容
  → 修正冲突点
  → 补充漏洞
  → 形成完整结论框架

顺序6：全员审阅
  → 共同审阅整合后的版本
  → 有异议则补充修正
  → 无异议则确认
```

```javascript
// 根据用户输入的议题特征，智能推荐模式
const recommendMode = (topic, constraints) => {
  const recommendations = [];

  // 检测关键词
  const isDecision = /决定|选择|方案|哪个好|取舍/.test(topic);
  const isCreative = /创意|想法|脑洞|创新|设计/.test(topic);
  const isDebate = /是否|能不能|应不应该|对错|优劣/.test(topic);
  const isReview = /评审|审查|PRD|文档|方案/.test(topic);
  const isComplex = /复杂|系统|多维度|综合/.test(topic);

  // 检测约束
  const isTwoPeople = constraints.participantCount === 2;
  const needsQuickConclusion = constraints.timeLimit !== undefined;

  // 推荐逻辑
  if (isReview) {
    recommendations.push({
      mode: 'specialized',  // 分工专项研讨式
      reason: 'PRD评审模式，适合方案审查',
      score: 95,
    });
  }

  if (isDebate) {
    if (constraints.participantCount >= 3) {
      recommendations.push({
        mode: 'standard-debate',  // 标准正反方辩论
        reason: '正式辩论，适合辨明对错',
        score: 90,
      });
    }
  }

  if (isCreative) {
    recommendations.push({
      mode: 'brainstorm',  // 头脑风暴
      reason: '创意发散，适合收集想法',
      score: 88,
    });
  }

  if (isDecision) {
    recommendations.push({
      mode: 'proposal-vote',  // 提案表决
      reason: '高效决策，适合多方案选择',
      score: 85,
    });
  }

  if (isTwoPeople) {
    if (isDebate) {
      recommendations.push({
        mode: 'qa-chase',  // 一问一答
        reason: '精准答疑，适合缩小信息差',
        score: 80,
      });
    } else {
      recommendations.push({
        mode: 'complement',  // 互补完善
        reason: '整合双方优势，适合深度商量',
        score: 85,
      });
    }
  }

  // 按分数排序
  return recommendations.sort((a, b) => b.score - a.score);
};
```

### 6.3 动态调整策略

```javascript
// 根据上下文动态调整输出深度
const dynamicDepthAdjustment = {
  // 当某一轮讨论特别深入时，自动扩展后续输出
  expandOnComplexity: (currentDepth, contextComplexity) => {
    if (contextComplexity > 0.8 && currentDepth === 'brief') {
      return 'normal';  // 自动升级
    }
    return currentDepth;
  },

  // 当用户时间有限时，自动精简
  compressOnTimePressure: (currentDepth, timeRemaining) => {
    if (timeRemaining < 30 && currentDepth === 'detailed') {
      return 'normal';
    }
    return currentDepth;
  },

  // 当讨论进入关键问题时，自动深化
  deepenOnKeyIssues: (currentDepth, isKeyIssue) => {
    if (isKeyIssue && currentDepth === 'normal') {
      return 'detailed';
    }
    return currentDepth;
  },
};
```

### 6.4 后端字数控制

```javascript
// 后端API层面的字数控制
const validateOutputLength = (output, expectedDepth) => {
  const length = output.length;
  const config = {
    brief: { min: 50, max: 150 },
    normal: { min: 200, max: 500 },
    detailed: { min: 800, max: 2000 },
  };

  const limits = config[expectedDepth];

  // 如果超出范围，进行截断或扩展
  if (length < limits.min) {
    return {
      valid: false,
      action: 'expand',
      suggestion: '内容偏少，建议补充更多细节',
    };
  }

  if (length > limits.max) {
    return {
      valid: true,  // 允许稍微超出
      action: 'truncate',
      suggestion: '内容偏多，建议精简',
    };
  }

  return { valid: true, action: 'none' };
};
```

### 8.3 讨论引擎改造

```javascript
// 统一的讨论引擎
class DiscussionEngine {
  constructor(config) {
    this.mode = config.mode;              // 讨论模式
    this.roles = config.roles;           // 角色配置
    this.depth = config.depth;            // 输出深度
    this.topic = config.topic;            // 议题
    this.flow = config.mode.flow;         // 流程配置
    this.currentStep = 0;                  // 当前步骤
  }

  // 执行下一步
  async nextStep() {
    if (this.currentStep >= this.flow.length) {
      return { done: true };
    }

    const step = this.flow[this.currentStep];
    const actor = this.getActor(step.actor);

    // 构建Prompt
    const prompt = this.buildPrompt(step, actor);

    // 调用AI
    const response = await this.callAI(prompt);

    // 广播消息
    this.broadcast('message', {
      step: step,
      actor: actor,
      content: response,
    });

    this.currentStep++;
    return { done: false, step, response };
  }

  // 根据步骤和角色构建Prompt
  buildPrompt(step, actor) {
    // 应用输出深度约束
    const depthPrompt = DEPTH_CONFIG[this.depth];

    // 获取角色Soul
    const soul = actor.soul;

    // 构建完整Prompt
    return `${soul}

议题：${this.topic}

${step.promptTemplate}

${depthPrompt.instruction}`;
  }
}
```
